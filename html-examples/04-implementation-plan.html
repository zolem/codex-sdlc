Phone-number client-side validation (US, SMS consent)

Add a shared Zod schema that strips formatting, validates US digits, and stores +1XXXXXXXXXX. Wire it into the existing StepPhone with real-time errors. No server change this slice; the client output matches the future HubSpot-oriented contract so backend can land independently.

Data flow

Solid path is the save-and-submit flow on Continue. Dashed clay is derived UI state — live error display and the Continue-button gate. The Zustand store is the single source of truth and only ever holds +1XXXXXXXXXX once the input is valid.

Decisions and the alternatives the architect rejected

Each row is a fork in the road. The "Considered" column is the alternative we are not taking — call it out if you disagree before phase 1 starts.

Decision Chosen Considered Why we chose it
Validation library Zod-only digit rules google-libphonenumber Already in use across the app for forms. Library used by 2FA but never wired to onboarding. Keeps bundle smaller and matches the team's existing pattern.
StepPhone form architecture Keep Zustand + local safeParse + phoneError state Refactor to useForm / ValidatedForm Onboarding is fully Zustand-driven. Introducing RHF for one field expands the diff and risks UX drift across steps. Same inline error UX is achievable without it.
When to normalize the stored value Normalize when parse succeeds (on change) Keep raw in store, normalize on blur only Requirements lean toward "stored value is E.164 when accepted." On-change normalization keeps the store invariant tight; blur-only adds a window where store ≠ contract.
Whitespace-only input Treat as empty (no error) Treat as invalid Optional field semantics. Product owner has not yet ruled — assumption documented; flip is one-line if they decide otherwise.
Schema composition z.preprocess + superRefine Plain regex · z.union of variants Preprocess gives us the strip + normalize transform; superRefine lets us emit distinct user-facing messages (non-numeric vs too short vs invalid).
Server validation Out of scope this slice Tighten recordPhoneConsent input now Backend → HubSpot integration ships separately. Tightening now risks rejecting legacy records mid-flight. Client emits the contract early so server can adopt safely later.

04

The one piece of code worth reading in full

Everything else is wiring. The schema below is the source of truth for the entire feature — sign off here is essentially sign off on the whole behavior.

// apps/web/src/utils/phone/us-sms-phone.schema.ts
import { z } from "zod";

// Output shape when accepted: E.164 US, exactly +1 followed by 10 digits.
export type UsSmsPhone = `+1${string}`;

const stripToDigits = (s: string) => s.replace(/\D/g, "");

export const optionalUsSmsPhoneSchema = z
  .string()
  .optional()
  .transform((raw) => (raw ?? "").trim())
  .superRefine((trimmed, ctx) => {
    if (trimmed === "") return;            // optional empty: pass
    const digits = stripToDigits(trimmed);
    if (digits.length === 0)
      ctx.addIssue({ code: "custom", message: "Please enter a phone number." });
    else if (digits.length < 10)
      ctx.addIssue({ code: "custom", message: "That number looks too short." });
    else if (digits.length === 11 && !digits.startsWith("1"))
      ctx.addIssue({ code: "custom", message: "We can't recognize that number." });
    else if (digits.length > 11)
      ctx.addIssue({ code: "custom", message: "That number has too many digits." });
  })
  .transform((trimmed): UsSmsPhone | undefined => {
    if (trimmed === "") return undefined;
    const digits = stripToDigits(trimmed);
    return (digits.length === 11 ? `+${digits}` : `+1${digits}`) as UsSmsPhone;
  });

// Used by both Continue gating and onboarding-flow save path.
export const isSmsPhoneStepComplete = (
  phone: string | undefined,
  consent: boolean,
): boolean => {
  if (!consent) return false;
  const result = optionalUsSmsPhoneSchema.safeParse(phone);
  return result.success && typeof result.data === "string";
};

06

Risks & open questions

SeverityRiskMitigation
Med Normalizing on every keystroke could feel jumpy as the user types. Only rewrite store when parse succeeds; raw input remains in the controlled value during typing.
Med Drift between Continue enabled and canGoNext() in flow-config. Single exported isSmsPhoneStepComplete() used by both, in phase 2 and 3.
Low Legacy prefill from phoneConsents may be non-E.164. Run the schema on prefill; normalize silently on load if valid, surface error otherwise.
Low Two phone validation stacks in repo (Zod here, libphonenumber in 2FA) could diverge. Document scope split in the schema file header. 2FA is product-distinct, not part of this slice.

Open questions blocking nothing — decide before phase 4

  1. Whitespace-only — empty or invalid? treat as empty
  2. onChange vs onBlur — which UX matches comparable fields? onChange, blur for parity if needed
  3. International / extension support — in this slice or follow-up? follow-up · US-only this slice
