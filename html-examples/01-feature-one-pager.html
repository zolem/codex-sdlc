Phone numbers will reject garbage and store a single clean shape

The SMS-consent phone field on onboarding (and the "finish your profile" flow that reuses it) will validate as the user types and save every accepted number in one format: +1XXXXXXXXXX. No more "abc", no more half-typed numbers, no more HubSpot rejects from inconsistent strings.

What's in scope

This release

  • Client-side Zod validation on the SMS phone field
  • Strip ( ) - space before checking
  • Normalize to E.164 +1XXXXXXXXXX on save
  • Reject non-numeric input and <10 digit numbers
  • Empty is fine — field stays optional
  • Shared schema used by onboarding and profile completion

Not in this release

  • Backend / HubSpot persistence
  • SMS delivery or OTP verification
  • Server-side format enforcement
  • International numbers or extensions

Why it matters

HubSpot drops numbers that don't match its SMS eligibility rules. Marketing has been cleaning up the spillover manually. This change moves the rejection to the moment the user is still in the form — they can fix the typo right there instead of getting a dead-end downstream.

The shape we'll save (+1 followed by 10 digits) is what the backend will expect when the HubSpot integration ships, so this slice is the contract being put in place early.

Risks we're tracking

Aggressive normalization on every keystroke

Could frustrate users mid-typing. Mitigation: only rewrite the stored value once the input is fully valid.

Whitespace-only behavior unclear

Treat it as empty (no error) or as invalid? Plan assumes empty unless product disagrees.

Legacy prefill from old consent records

Some users may have non-E.164 numbers stored. Plan: parse on load and normalize once seen.

Open questions for product

  1. Whitespace-only — empty or invalid?
  2. Validate on every keystroke, or only on blur?
  3. Should international numbers / extensions be in this slice, or a later one?

Engineer can proceed under the documented assumptions; product can override before merge.
